# M4 - Mainnet Production Executor

Status: In progress  
Source of truth: `.ai/specs/solana-signal-bot-spec-v2.md` sections 3, 5.2, 5.5, 6.1, 7/M4, and 8.

## Goal

Turn the M3 RPC executor into a production-grade mainnet executor, still RPC-only and explicitly not Jito. M4 proves that the bot can quote, build, simulate, sign, submit, confirm, reconcile, persist, and expose metrics for tiny real mainnet trades without double-spend behavior.

## M4 Scope

Must implement:
- ALT hydration for Jupiter lookup tables.
- Versioned transaction compilation.
- Helius priority fee API via `getPriorityFeeEstimate`.
- Two-pass compute unit simulation with 1,400,000 CU placeholder, then `ceil(unitsConsumed * 1.15)`.
- RPC-only submission with `skipPreflight: true` and `maxRetries: 0`.
- Confirmation polling that returns `confirmed`, `failed_onchain`, `expired`, or `uncertain`.
- Post-trade reconciliation using `getTransaction(signature, { maxSupportedTransactionVersion: 0 })`.
- Trade persistence with actual output amount when confirmed.
- `/metrics` populated for all required M4-relevant counters, histograms, and gauges.
- Dry-run behavior immediately before submission, inside the executor path.
- Deterministic tests first, then implementation.
- Guarded live mainnet micro-trade harness for final acceptance evidence.

Must not implement in M4:
- Jito bundles, Jito tip transactions, or Jito fallback logic. That is M5.
- Sells, position management, TP/SL, dashboard, or multi-wallet behavior.
- Jupiter `/swap`; only `/swap-instructions` is allowed.
- SDK-level retries. `maxRetries` must stay `0`.
- Automatic retry after `uncertain`. It is a terminal human-intervention state.

## Existing Baseline

Already present as of 2026-04-29:
- `src/executor/index.ts` orchestrates quote -> swap instructions -> build -> sign -> RPC submit -> confirm -> trade write.
- `src/executor/index.ts` uses versioned transactions through `@solana/kit`.
- `src/executor/index.ts` already fetches lookup table addresses and compresses the transaction message.
- `src/executor/index.ts` uses fixed CU limit `1_400_000` and fixed CU price `5_000n`; these must be replaced by M4 logic.
- `src/executor/index.ts` has deterministic confirmation polling coverage for confirmed, expired, pre-submit failure, failed-onchain, and uncertain outcomes.
- `src/metrics/registry.ts` already defines the required metric names.
- `src/risk/blockers.ts` already populates wallet/daily-spend/kill-switch gauges.

## Suggested Implementation Sequence For Future LLMs

1. Start with tests around the current executor seams; do not run live mainnet first.
2. Extract small executor modules only as needed to make priority fee, compute simulation, confirmation, and reconciliation testable.
3. Replace fixed compute/fee constants after tests exist.
4. Add reconciliation after confirmed-state persistence is stable.
5. Add dry-run before live harness so mainnet config can be checked without spending.
6. Only then run guarded live micro-trades and record evidence.
